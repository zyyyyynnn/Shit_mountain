# 安全政策

Shit Mountain 用于教育、娱乐和代码审查训练。项目会展示安全反模式，但不接受可直接伤害真实系统或用户的内容。

## 报告仓库自身的安全问题

请不要为以下问题公开创建 Issue：

- GitHub Actions 权限或令牌泄漏；
- 可以在维护者上下文中执行任意代码的漏洞；
- 分支清理、发布或自动化中的权限绕过；
- 仓库中意外提交的真实凭证、个人信息或第三方私有代码。

请通过仓库所有者 GitHub 主页提供的私密联系方式报告，并包含：受影响文件或工作流、复现条件、潜在影响和建议修复方式。不要在报告中包含不必要的真实数据。

## 展品安全边界

允许：

- 使用虚构数据展示代码异味和安全反模式；
- 在隔离、无害的示例中解释错误处理、输入校验或权限设计问题；
- 展示预期失败，但必须由 `exhibit.json` 明确声明。

禁止：

- 真实密钥、访问令牌、个人信息或私有源码；
- 恶意软件、持久化、凭证窃取或破坏性命令；
- 针对真实服务的攻击脚本、扫描器或可武器化利用链；
- 无限循环、fork bomb、磁盘填充、内存耗尽等资源消耗攻击；
- 未经授权的第三方代码或事故材料。

## 执行模型

展品执行器提供临时目录、环境变量清理、超时和输出限制，但它不是操作系统级沙箱，也不承诺阻断网络。所有可执行展品仍必须经过人工审查。

CI 对普通检查使用只读仓库权限。任何需要写权限的工作流必须与展品执行分离，并限制触发条件。
