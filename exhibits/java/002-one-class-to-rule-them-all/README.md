# 002：One Class to Rule Them All

> 王座厅里只有一个类。用户、库存、定价、支付、邮件、审计和老板的临时要求，都要先向它行礼。

## 案发现场

`EverythingManagerFinalV2` 最初只是一个“统一处理订单”的便利入口。随着需求增加，库存字段、会员折扣、支付路由、邮件状态、审计计数和管理层特批全部被塞进同一个对象。

它仍然能够运行，但每项新需求都会迫使开发者重新进入同一座王座厅，触碰彼此无关的状态和分支。

## 部门兼并记录

这个类同时承担：

- 用户输入校验；
- 订单编号生成；
- 库存查询与扣减；
- 会员定价和运费规则；
- 支付通道路由；
- 支付异常处理；
- 邮件发送状态；
- 审计计数；
- 老板临时免运费特批；
- 最终输出格式拼装。

代码中的“为了方便，先放这里”并不是一句装饰性笑话，而是 God Object 最常见的形成机制：每次局部便利都把更多职责永久并入中央对象。

## 主要罪名

### 1. 共享可变状态形成宫廷政治

`stock`、`auditCount`、`currentUser`、`lastPayment`、`lastEmail` 等字段由同一方法按隐式顺序修改。调用者无法只使用定价或支付能力，也很难判断某个失败分支留下了哪些状态。

### 2. 业务规则与基础设施互相绑架

折扣、库存、支付、通知和审计本应拥有不同的变化原因。现在修改信用卡审核阈值，也必须在同一个类中穿过库存和邮件逻辑。

### 3. 测试只能召见整座帝国

要测试一个折扣分支，必须同时构造库存、支付类型、用户、邮件状态和审计状态。测试无法替换单个部门，也无法隔离失败来源。

### 4. 隐式调用顺序成为隐藏 API

必须先计算价格，再支付，再扣库存，再发邮件，最后记审计。这个顺序没有由类型或接口表达，只存在于一个超长方法的阅读体验里。

## 爆炸姿势

假设新增“支付成功后邮件失败不得扣库存”的要求：

1. 需要修改支付、库存、邮件和审计四段代码；
2. 必须重新决定失败时哪些字段回滚；
3. 可能破坏老板免运费特批；
4. 现有测试很难指出是哪项职责发生回归；
5. 下一位维护者会继续在同一方法里添加补丁，因为拆分成本越来越高。

God Object 的危险不只是文件很长，而是**不同变化原因被迫共享同一个修改入口**。

## 帝国解体方案

`fixed/` 将职责拆分为：

| 部门 | 责任 |
|---|---|
| `OrderService` | 编排订单流程，不实现基础设施细节 |
| `InventoryService` | 校验与扣减库存 |
| `PricingService` | 计算折扣和运费 |
| `PaymentService` | 处理支付规则并返回明确结果 |
| `NotificationService` | 发送订单通知 |
| `AuditLog` | 独立维护审计记录 |
| `OrderRequest` / `OrderReceipt` | 用数据结构表达输入与输出合同 |

修复版没有引入框架，也没有把所有东西抽象成接口。它只建立足够清晰的职责边界，让变化能够被定位、测试和替换。

## 行为合同

两个版本对同一订单必须输出：

```text
ORDER-1001|PAID|stock=7|mail=sent|audit=1
```

这项校验防止“帝国解体”借机修改原有业务结果。

## 运行方法

在仓库根目录执行：

```bash
bash scripts/validate.sh
```

也可以单独编译后运行：

```bash
java EverythingManagerFinalV2
java CompanyOperations
```

## 屎山指数

事故现场当前得分为 **100 / 100**，主要贡献来自决策点、魔法数字、全局可变状态、超长方法和重复语句。

这个分数用于解释代码异味，不用于评价写下这段代码的人。真实 God Object 通常不是某个人一次写成的，而是组织压力和连续局部妥协长期沉积的结果。
