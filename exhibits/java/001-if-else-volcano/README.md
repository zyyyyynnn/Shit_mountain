# 🌋 展品 001：If-Else 火山

> 风险等级：**熔岩正在会议室门口流动**

## 案发现场

某电商系统需要根据用户类型、节日状态和优惠券计算最终价格。第一版只有两个条件，于是开发者非常自信地说：

> “这个逻辑很简单，直接写几个 if 就行。”

后来需求没有停止，if 也没有停止。

## 展区结构

```text
001-if-else-volcano/
├── bad/ShitDiscountCalculator.java
└── fixed/CleanDiscountCalculator.java
```

两个版本对示例输入都返回 `70`。修复版的目标不是改变业务结果，而是让下一个需求不必穿防火服。

## 主要罪名

### 1. 嵌套式地质增生

用户类型、价格、节日状态互相嵌套。每增加一个条件，都可能产生新的分支组合。

### 2. 字符串充当类型系统

`"VIP"` 和 `"NORMAL"` 可以被拼错，也可能收到 `null`，问题直到运行时才出现。

### 3. 魔法数字自然保护区

`100`、`30`、`20`、`5`、`3`、`1`、`2`、`999` 都没有业务名称。考古人员只能通过猜测恢复产品规则。

### 4. 优惠券靠复制粘贴繁殖

每新增一种券，就继续追加一个独立 `if`。互斥关系完全依赖开发者的记忆。

### 5. 谎言注释

`// 简单处理一下` 没有解释规则，只是在事故现场留下了一张乐观纸条。

## 它会怎么炸？

以下需求任意出现两个，原实现就会迅速隆起：

- 增加 `SUPER_VIP` 和员工类型。
- 优惠券改为互斥、叠加或按比例计算。
- 节日按地区生效。
- 折扣需要记录命中的规则。
- 价格不能低于某个商品的最低售价。

## 铲屎路线

修复版采用小步重构：

1. 用 `enum UserType` 代替自由字符串。
2. 将会员折扣和优惠券折扣拆成独立函数。
3. 使用 `switch` 显式表达互斥规则。
4. 将最终价格下限集中处理。
5. 保留遗留优惠券 `999` 的行为，避免“重构顺便改业务”。

它不是终极架构。规则继续增长时，可以进一步引入规则对象或决策表；当前版本只负责把火山从“随时喷发”降级为“可测量地热活动”。
